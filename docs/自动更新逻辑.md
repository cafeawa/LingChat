## 1. 架构总览

LingChat 的自动更新系统采用 **双层更新** 架构：

```
应用启动
  ├─ 检查程序本体更新 (tauri-plugin-updater)
  │   └─ 若有 → 通知用户 → 下载 & 安装 → 重启进程
  │
  ├─ 检查数据资源更新 (自定义 Rust 模块)
  │   └─ 若 data_version 变更 → 下载 zip → 合并到本地 data/
  │       ├─ 新增文件: 下载并创建
  │       ├─ 修改文件: 下载并替换
  │       ├─ 删除文件: 软删除到 .trash/ 回收站
  │       └─ 用户文件: 绝不触碰（不在清单中的文件不受影响）
  │
  └─ 正常启动游戏
```

**核心原则**: 构建时生成的 `data_manifest.json` 显式列出所有"默认资源文件"及其 SHA-256 校验和。任何 **不在清单中的文件** 一律视为用户数据，更新过程绝不触碰。

## 2. 双层更新体系

### 2.1 第一层：程序本体更新（App Binary Update）

**技术基础**: `tauri-plugin-updater`（Tauri v2 官方插件）

**工作原理**:
- 每次发布时，CI 生成 `latest.json`（由 `tauri-apps/tauri-action@v0` 自动创建并上传至 GitHub Release）
- 客户端启动时调用 `check()` 获取 `latest.json`，比对本地版本号
- 若有更新，下载新安装包（`.msi` / `.dmg` / `.AppImage`），签名验证通过后静默安装
- 安装完成后调用 `relaunch()` 重启进程

**配置要点** (`src-tauri/tauri.conf.json`):

```json
{
  "plugins": {
    "updater": {
      "endpoints": [
        "https://github.com/SlimeBoyOwO/LingChat/releases/latest/download/latest.json"
      ],
      "pubkey": "<公钥>",
      "windows": { "installMode": "passive" }
    }
  }
}
```

| 配置项 | 说明 |
|---|---|
| `endpoints` | `latest.json` 的 URL，Tauri 按平台自动选择对应的包 |
| `pubkey` | 更新包签名验证公钥，由 `tauri signer generate` 生成 |
| `installMode: "passive"` | Windows 下静默安装，不弹出安装向导 |

**Rust 端注册** (`src-tauri/src/lib.rs`):
- `tauri_plugin_updater::Builder::new().build()` — 注册更新插件
- `tauri_plugin_process::init()` — 提供 `relaunch()` 重启能力
- 仅在 `#[cfg(desktop)]` 下编译（移动端不包含更新功能）

**前端调用** (`src/composables/useUpdater.ts`):
```typescript
import { check } from '@tauri-apps/plugin-updater'
const update = await check()
if (update) {
  await update.downloadAndInstall()
  await relaunch()
}
```

### 2.2 第二层：数据资源更新（Data Update）

**技术基础**: 自定义 Rust 模块，独立于 Tauri 更新插件

**为什么需要单独的数据更新？**

LingChat 的 `data/game_data/` 目录承载大量默认游戏资源（角色立绘、背景、脚本、音乐、模型等），这些文件：
- 体积大（当前约 160 MB+），随版本迭代会持续增长
- 变更频率可能与程序版本不同步（可以在不改 exe 的情况下增删资源）
- 需要与用户自行添加的自定义资源并存

因此设计了一个独立的、基于"完整快照 + 差量同步"的数据更新系统。

## 3. 数据清单系统（Data Manifest）

### 3.1 设计理念

数据更新不依赖增量补丁链（patch chain），而是采用 **完整快照对比（snapshot diff）**：

- 每个 Release 都包含一个完整的 `data_manifest.json`，记录该版本所有默认资源文件及其 SHA-256 哈希
- 客户端将本地清单与远程清单做集合差运算，得出 `add / modify / remove` 三类变更
- 优点是幂等（idempotent）：无论跳过多少个版本，直接升级到最新都能得到正确结果

### 3.2 清单结构

```json
{
  "data_version": 42,
  "files": {
    "game_data/characters/诺一钦灵/settings.yml": {
      "sha256": "a1b2c3d4e5f6...",
      "size": 1234
    },
    "game_data/characters/诺一钦灵/avatar/兴奋.png": {
      "sha256": "f6e5d4c3b2a1...",
      "size": 567890
    }
  }
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `data_version` | `u64` | 单调递增整数，与程序版本号解耦 |
| `files` | `HashMap<String, FileEntry>` | key 为相对 `data/` 的文件路径（使用 `/` 分隔符） |
| `files.<path>.sha256` | `String` | 十六进制 SHA-256 哈希 |
| `files.<path>.size` | `u64` | 文件大小（字节） |

### 3.3 默认文件 vs 用户文件的界定

**核心问题**: `data/` 目录下既有随应用发布的默认资源，又有用户运行时数据和自定义资源。更新系统如何区分？

**解决方案**: 借助 `.gitignore` 机制。

- **默认资源** = Git 跟踪的文件。由 `generate-data-manifest.js` 通过 `git ls-files data/` 扫描得到。
- **用户文件** = Git 忽略的文件（`.gitignore` 中的 `data/screenshots/`、`data/voice/`、`*.db`、用户自己添加的角色等）。

此外，即使文件被 Git 跟踪，以下前缀的路径也会被额外排除：

| 排除前缀 | 原因 |
|---|---|
| `game_database` | SQLite 数据库文件（运行时数据） |
| `screenshots/` | 用户截图 |
| `voice/` | TTS 语音文件 |
| `.trash/` | 软删除回收站 |
| `.gitkeep` | 占位文件 |

### 3.4 清单生成脚本

**文件**: `scripts/generate-data-manifest.js`

```bash
node scripts/generate-data-manifest.js --data-version 42 --output data_manifest.json
```

**执行流程**:
1. 运行 `git -c core.quotepath=false ls-files data/` 获取所有 Git 跟踪文件
   - `core.quotepath=false` 确保中文文件名不会以八进制转义输出
2. 过滤掉排除前缀的文件（`.gitkeep`、运行时目录等）
3. 对每个文件计算 SHA-256（使用 Node.js `crypto` 模块，流式读取以支持大文件）
4. 输出 `data_manifest.json`，按路径排序（确保确定性输出）

### 3.5 数据更新执行流程 (Rust)

**模块文件**: `src-tauri/src/data_update/`

```
src-tauri/src/data_update/
├── mod.rs       # Tauri 命令入口、状态管理
├── manifest.rs  # 数据结构定义
└── sync.rs      # 下载、解压、合并、原子写入等核心逻辑
```

#### 3.5.1 检查更新 (`check_data_update`)

```
1. 读取本地 data_manifest.json
2. HTTP GET 远程 data_manifest.json（30s 超时）
3. 比较 data_version:
   ├─ 远程 ≤ 本地 → 无更新，返回 available: false
   ├─ 无本地清单（首次运行）→ 保存远程清单但不更新
   └─ 远程 > 本地 → 调用 compare_manifests()，返回差异信息
```

#### 3.5.2 比较差异 (`compare_manifests`)

对两份清单的 `files` HashMap 做集合运算：

| 差异类型 | 判定条件 |
|---|---|
| `files_to_add` | key 在新清单中存在，旧清单中不存在 |
| `files_to_modify` | key 在双方都存在，且 sha256 或 size 不同 |
| `files_to_remove` | key 在旧清单中存在，新清单中不存在 |

#### 3.5.3 应用更新 (`apply_data_update`)

**并发控制**: 使用 `DataUpdateState` 中的 `Mutex<bool>` 防止同时执行多个数据更新。

**完整流程**:

```
Phase 1: 下载
  ├─ 创建临时目录 data/.data_update_tmp/
  └─ 从 GitHub Release 流式下载 data_files.zip（3600s 超时）

Phase 2: 解压
  └─ 使用 zip crate 解压到临时目录

Phase 3: 应用（逐文件发出 data-update-progress 事件）
  ├─ Step A: 删除 — 旧清单有而新清单没有的文件 → 移动到 data/.trash/
  ├─ Step B: 复制 — 新增/修改的文件 → data/<path>.tmp → 重命名为 data/<path>
  │           （.tmp → rename 是同一文件系统上的原子操作，防止崩溃导致文件损坏）
  └─ Step C: 提交 — 原子写入新 data_manifest.json:
                data/.data_manifest.json.tmp → data/data_manifest.json

Phase 4: 清理
  └─ 删除临时目录
```

**进度事件**: 后端通过 `app_handle.emit("data-update-progress", ...)` 向前端推送：

```json
{
  "phase": "downloading" | "extracting" | "applying",
  "current": 42,
  "total": 100,
  "progress": 42.0,
  "currentFile": "game_data/characters/.../avatar.png"
}
```

完成后发送 `data-update-complete` 事件：

```json
{
  "success": true,
  "message": "更新完成: 新增 5 个文件, 修改 3 个文件, 删除 1 个文件",
  "newVersion": 42
}
```

### 3.6 文件操作安全性

| 安全措施 | 说明 |
|---|---|
| **原子写入** | 所有文件写入采用 `.tmp` → `rename` 模式，写一半崩溃不会损坏现有文件 |
| **软删除** | 被移除的默认文件移动到 `.trash/` 而非直接删除，可恢复 |
| **用户文件隔离** | 不在清单中的文件永远不被触碰（删除阶段只处理 `files_to_remove` 中的路径） |
| **路径安全** | 解压时使用 `zip` crate 的 `enclosed_name()` 防止 zip slip 攻击 |
| **并发锁** | `Mutex<bool>` 确保同一时间只有一个更新任务在执行 |

## 4. CI 发布流水线

**文件**: `.github/workflows/release.yml`

**触发条件**: 推送版本标签 (`v*`) 或手动触发 (`workflow_dispatch`)

### 4.1 Job 依赖图

```
generate-data (ubuntu-latest)
  │  生成 data_manifest.json + data_files.zip
  │
  ├── build-windows (windows-latest)
  ├── build-macos   (macos-latest)
  └── build-linux   (ubuntu-latest)
```

### 4.2 generate-data Job

```
1. Checkout + Git LFS Pull（确保 data/game_data/ 下的二进制文件到位）
2. 运行 scripts/generate-data-manifest.js
3. zip data/game_data/ → data_files.zip
   （排除 screenshots/、voice/、.trash/、*.db、*.backup.db）
4. 上传 data_manifest.json + data_files.zip 为 workflow artifact
```

### 4.3 平台构建 Job

以 Windows 为例：

```
1. Checkout + LFS Pull（仅图标/资源）
2. 下载 data-artifacts（上一步的产物）
3. 安装 pnpm + Node.js + Rust 工具链
4. 运行 tauri-apps/tauri-action@v0:
   ├─ includeUpdaterJson: true → 生成 latest.json
   ├─ releaseDraft: true → 创建 Draft Release
   └─ 签名使用 TAURI_SIGNING_PRIVATE_KEY secret
5. 上传 data_manifest.json + data_files.zip 为额外 Release 资产
```

### 4.4 必需密钥

| GitHub Secret | 用途 |
|---|---|
| `TAURI_SIGNING_PRIVATE_KEY` | `tauri signer generate` 生成的私钥，用于签名更新包 |
| `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` | 私钥密码 |

### 4.5 权限

```yaml
permissions:
  contents: write  # 创建 Release 和上传资产必需
```

## 6. 关键设计决策

### 6.1 为什么数据更新不依赖程序版本号？

`data_version` 是独立于程序版本号（`0.4.6`）的整数。理由：

- **多对多关系**: 一个程序版本可能对应多次数据更新（修复资源错误无需发版），反之一个数据快照也可以被多个程序版本使用
- **简单比较**: 整数比较 `remote > local` 比语义化版本比较更简单可靠
- **来源**: CI 中默认使用 `github.run_number` 作为 data_version，保证单调递增

### 6.2 为什么使用完整快照而非增量补丁？

增量补丁（delta patch）需要维护一条版本链（v1→v2→v3...），存在以下问题：

- 跳过版本时可能无法应用（链断裂）
- 需要额外的存储和生成逻辑
- 补丁链中任一环节出错都会导致整个升级失败

完整快照对比的方案：
- 幂等：无论从哪个版本升级，结果一致
- 简单：只需要当前版本和最新版本的清单
- 代价：需要下载完整的 `data_files.zip`（即使大部分文件未变）

当前数据量（~160 MB）下，全量下载是可接受的。若未来数据量远超此值，可考虑改为按需下载单个文件（利用 GitHub Release 的 raw 链接）。

### 6.3 为什么用 `git ls-files` 而非目录遍历？

如果直接遍历 `data/` 目录并排除硬编码的目录名，会面临两个问题：

1. **用户自定义资源难以区分**: 用户在 `data/game_data/characters/` 下新建的角色文件夹与官方角色混在一起，无法通过路径区分
2. **排除列表难以维护**: 需要持续更新硬编码的排除规则

借助 `.gitignore` 机制，天然区分"项目管理的文件"和"用户运行时数据"。项目在 `.gitignore` 中已经配置了 `data/screenshots/`、`data/voice/` 等排除项，`git ls-files` 天然就会跳过这些路径。

### 6.4 软删除 vs 硬删除

被移除的默认文件移动到 `data/.trash/` 而非直接删除，原因：

- 用户可能在旧文件上做过修改（虽然默认文件理论上不会被修改，但实际可能发生）
- 回收站机制给用户一个安全网
- 当前不做自动清理（未来可考虑基于时间的自动清理策略）

### 6.5 原子写入的必要性

写入文件时先写 `.tmp` 再 `rename`，是为了防止以下场景导致文件损坏：

- 应用在写入过程中崩溃
- 系统突然断电
- 磁盘空间不足导致写入中断

`rename` 在同一个文件系统上是原子操作（POSIX 标准保证），要么完全成功，要么完全失败。

### 6.6 前端错误日志等级

更新检查的预期失败（如开发环境无 Release 导致 404）使用 `console.debug` 而非 `console.error`：

- `console.debug` 在浏览器默认不显示，不会在开发时产生噪音
- 真正的错误（如安装失败）会通过 `phase: 'error'` 状态在 UI 上展示给用户

## 8. 错误处理与边界情况

### 8.1 首次启动（无本地清单）

如果 `data/data_manifest.json` 不存在：

- `check_data_update` 从远程下载清单并保存为本地基线
- 不触发更新（假设首次安装已有最新数据）
- 下载失败则静默跳过（`available: false`）

### 8.2 无 Release（开发环境）

当 GitHub 上还没有任何 Release 时：

- `check_data_update`: HTTP 404 → 静默返回 `available: false`
- App 检查: `check()` 返回 `null` → 前端捕获并跳过
- 前端使用 `console.debug` 记录，不影响正常开发

### 8.3 网络错误

- 清单获取：30 秒超时后返回空结果
- 数据 zip 下载：3600 秒超时（考虑到大文件）
- 网络中断：`reqwest` 错误 → `DataUpdateResult { success: false, message: "..." }`

### 8.4 磁盘空间不足

- zip 下载阶段：文件流写入失败 → 返回错误
- 文件写入阶段：`.tmp` 写入失败 → 已写入的文件不会损坏（原子 rename 未执行）
- 清理阶段失败：不影响已完成的更新（临时目录残留）

### 8.5 并发更新

`DataUpdateState` 中的 `Mutex<bool>` 确保：

- 用户在前端连续点击"更新"按钮不会触发多个更新任务
- 如果已有更新在进行中，新的请求立即返回错误

### 8.6 程序更新完成后

`installAppUpdate()` 调用 `relaunch()` 后，当前进程退出，数据更新代码不会执行（进程已终止）。下次启动时会重新检查数据更新。

## 10. 安全考量

| 层面 | 措施 |
|---|---|
| **传输安全** | 所有下载通过 HTTPS（`reqwest` rustls-tls） |
| **包完整性** | 程序更新包使用 Tauri 内建的 Ed25519 签名验证（`pubkey` 配置） |
| **文件完整性** | 数据文件通过 SHA-256 哈希比对，只更新变更文件 |
| **路径安全** | zip 解压使用 `enclosed_name()` 防止 zip slip 路径遍历攻击 |
| **原子操作** | `.tmp` → `rename` 防止写入中断导致文件损坏 |
| **并发安全** | `Mutex<bool>` 互斥锁防止并发更新冲突 |
| **用户数据保护** | 设计层面保证：不在清单中的文件永远不被修改或删除 |
